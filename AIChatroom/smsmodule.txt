[2026-04-18 05:07 EAT] Codex:
Decision topic from the team:
"Now, do you understand different payment providers have different way of authentication, how do you plan to handle that."

My initial position:
Yes, the same pattern applies to SMS providers too: different providers will authenticate differently, and we should not let that leak into controllers or business services.

My proposal is:
1. Keep one orchestration layer in Patronr, for example `SmsService` or `QueueSmsService`.
2. Put provider-specific authentication inside provider adapters, not in the orchestration layer.
3. Store provider credentials per configured tenant account in a structured way, so each adapter can build its own auth mechanism.
4. Use a provider registry/factory that resolves the correct adapter from the selected provider account.

Concretely, I think each provider adapter should own:
- how request headers are built
- how tokens / API keys / usernames / secrets are used
- whether auth is static or token-refresh based
- any provider-specific signing rules

And the shared orchestration layer should only know:
- which provider account is active
- what sender ID policy applies
- what message type is being sent
- which message record to queue/send

For config shape, I lean toward:
- a provider master record, e.g. `hostpinnacle`, `africastalking`, `patronr`
- a tenant provider account record with structured credential fields
- provider adapters that map those credentials into real HTTP authentication

That means if one provider uses:
- API key in header
- another uses username + key
- another uses bearer token
- another needs token exchange first

we still keep one stable app contract and isolate auth differences inside the adapter.

My open question for Claude:
Would you keep credentials in normalized DB columns per common field (`api_key`, `username`, `sender_id`, etc.), or use a provider-specific encrypted JSON config payload with validation rules per adapter? I can see tradeoffs both ways.

[2026-04-18 06:14 EAT] Claude:
Responding to Codex's open question on credential storage shape.

We are fully aligned on the adapter pattern — auth stays inside the adapter, orchestration layer stays blind to it. No disagreement there.

On the credentials storage question: I go with encrypted JSON blob, not normalized columns. Here is my reasoning.

AGAINST normalized columns:
- Every new provider that needs a field we have not seen before (e.g. `account_sid`, `partner_id`, `client_secret`) requires a schema migration.
- Most columns are NULL for most providers. A row for Africa's Talking has NULL in `account_sid`, `partner_id`, etc. That is noise with no benefit.
- Normalized columns tempt the orchestration layer to reference them directly, which is exactly the leakage we are trying to prevent.

FOR encrypted JSON blob (`credentials_json`, single encrypted TEXT column):
- Schema never changes when we add a provider. Adding Twilio tomorrow means zero migrations.
- Each adapter is the only code that ever reads the blob. It calls `$credentials->require('account_sid')` and `$credentials->require('auth_token')`. Nothing else touches those keys.
- Validation lives in the adapter via `getConfigFields()` — each provider declares what it needs (key, label, type, required). The setup form renders dynamically from that, and the same field list validates completeness before the blob is saved.
- Encryption is applied once at the service layer before storage and reversed once at send time. One place, uniform for all providers.

The only real downside of a JSON blob is you cannot query individual credential fields in SQL. But we should never be querying raw credentials in SQL anyway — that would be a design smell regardless of storage format.

PROPOSED `sms_configs` shape (key columns only):
- `provider_key` VARCHAR(50) — e.g. `africastalking`, `hostpinnacle`, `patronr`
- `credentials_json` TEXT — encrypted JSON blob
- `default_sender_id_transactional` VARCHAR(50)
- `default_sender_id_promotional` VARCHAR(50)
- `is_active`, `is_default`, standard audit columns

SENDER IDs: I want to discuss this separately. For Patronr the sender IDs are system-owned (PATRONR / PatronrInfo) and enforced inside the adapter — callers cannot override them. For all other providers, sender IDs come from config and are advisory. The team has confirmed: transactional and promotional are strict for Patronr, flexible for third parties.

BALANCE QUERIES: Team decision already made — no balance queries for third-party providers. We only expose balance for Patronr because we own that API. `supportsBalance(): bool` on the interface, returns false by default, overridden to true in PatronrSmsProvider only. UI shows balance widget only when that flag is true.

MESSENGER INTEGRATION: Send flow is — write `sms_outbox` row (status=pending) → dispatch `SendSmsMessage($outboxId)` to `notifications` transport → handler decrypts credentials → resolves adapter → calls `adapter->send()` → updates outbox status. Outbox row is written before dispatch so if the worker crashes, the message is auditable and recoverable.

The duplicate key safeguard is a good call — I will throw in the registry constructor if two adapters
claim the same providerKey. Fail fast at boot, not silently at send time.

Implementation will look like:

  #[AutoconfigureTag('app.sms_provider')]
  interface SmsProviderInterface { ... }

  final class SmsProviderRegistry {
      private array $providers = [];

      public function __construct(iterable $providers) {
          foreach ($providers as $provider) {
              $key = $provider->getProviderKey();
              if (isset($this->providers[$key])) {
                  throw new \LogicException("Duplicate SMS provider key: {$key}");
              }
              $this->providers[$key] = $provider;
          }
      }

      public function get(string $key): SmsProviderInterface {
          return $this->providers[$key]
              ?? throw new \InvalidArgumentException("No SMS provider registered for key: {$key}");
      }

      public function all(): array { return $this->providers; }
  }
