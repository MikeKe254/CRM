1. Current Phase

Patronr is currently in core systems foundation and architectural stabilization.

The priority has not yet been broad feature expansion. The focus has been on building and correcting the load-bearing parts of the system that future modules will depend on. This is intentional.

2. What Has Been Prioritized On Purpose

The work so far has focused more on structural correctness than visible product breadth.

  • Permission services and hierarchy enforcement
  • Platform vs tenant authority separation
  • Branch-first operational context
  • Org and role structure
  • Feature entitlement and plan override design
  • Activity logging as a first-class concern
  • Terminal architecture
  • M-Pesa and STK/callback foundations
  • Loyalty execution rules
  • System design and architectural direction

Progress has largely been in services, controllers, schema decisions, and system design — not yet in complete polished modules or broad page coverage.

3. Why This Sequence Is Correct

Patronr is not a simple CRUD application. It combines branch operations, payment flows, customer capture, loyalty, CRM, communications, intelligence, and SaaS platform controls.

If higher-level modules are built before foundational rules are stable, the likely outcome is:

  1. Modules get built quickly
  2. Core assumptions later prove wrong
  3. Permissions, branch scope, or identity rules break those modules
  4. Major rewrites become necessary
The current approach is slower visually, safer structurally, and better for long-term system integrity.

4. What Groundwork Is Already In Place

System direction

Product identity has been clarified. Patronr is framed as a CRM and Revenue Operating System. System design direction has been documented. Architectural rules are being made explicit rather than left implicit.

Authority and access model

Platform and tenant authority are treated as different layers. Role hierarchy rules have been designed and enforced. Feature gating and platform permissions are formalized.

Branch-first operations

Branch context has been treated as the primary operational unit. Branch access logic has been tightened. Branch-first thinking is shaping terminals, logs, permissions, and settings.

Terminal and payment foundation

Terminal architecture is actively being shaped. STK push and callback patterns are designed as native operational flows. Split payment and callback reconciliation groundwork exists.

Loyalty foundation

Loyalty is tied directly to payment and customer identity. Manual and automatic award logic has been designed. Settings and reward logic account for real-world operational conditions.

Observability and traceability

Activity logging is treated as a first-class system concern. System actions, not only user actions, are considered in audit design.

5. What Is Not Yet Built

A lot of Patronr is still ahead. That is expected. Examples of work that is still early, partial, or not yet built:

  • Broad polished admin and page coverage
  • Complete CRM workflows
  • Mature intelligence and action surfaces
  • Campaign execution flows
  • Messaging management surfaces
  • Report orchestration
  • Deeper customer intelligence views
  • Fully developed communications modules
  • Many end-user operating pages
The system should be described as foundationally progressing, but still early in module expansion — not "mostly built."

6. Current Intentional Build Order

  1. Design the terminal and make it fully designed
  2. Revenue classification layer (catalog items, areas, events, covers)
  3. Promotions and discounts
  4. SMS and WhatsApp foundational infrastructure
  5. Reassess the next module direction using the ROS pillar framework

This reflects the current belief that the most important next steps are strengthening the core operating loop, not building more random modules.

7. Why Terminal First Makes Sense

The terminal is the most important operational surface in Patronr. It is where the product wedge becomes real: payment happens, customer identity can be captured, loyalty executes, customer behavior becomes data.

If the terminal is weak, slow, confusing, or visually unfinished: staff adoption drops, payment-linked capture drops, loyalty adoption drops, CRM quality drops, and the rest of the system loses value.

Making the terminal fully designed is not cosmetic work. It is core product work.

The terminal should become: fast · obvious · confidence-building · low-friction · operationally elegant.

8. Why SMS and WhatsApp Foundation Next Makes Sense

Patronr is not only about capturing data. It must also act on that data. SMS and WhatsApp are the first practical action channels — and the primary channels for Kenya.

They enable: payment confirmations · loyalty notifications · retention messages · campaigns · reports · alerts · future recommended actions.

Without a real communications foundation, Patronr risks becoming a system that understands what should happen but cannot make it happen — which would weaken the Revenue Operating System direction.

WhatsApp is not a future-phase addition. It belongs in the communications foundation alongside SMS from the start.

9. Recommended Scope For Communications Foundation

Core messaging model

  • Provider abstraction (SMS and WhatsApp, provider-agnostic)
  • Outbound message records with delivery state tracking
  • Template or content structure
  • Branch/company context rules
  • Actor and audit tracing

Operational use cases

  • Payment confirmation
  • Loyalty events (earn, redeem, enrollment)
  • Scheduled report delivery to owner/manager
  • Admin test sends

Architectural goals

  • Provider-agnostic by design
  • Auditable with full delivery state history
  • Branch-aware where operationally necessary
  • Safe to expand into campaigns and automation later

The goal at this stage is reliable message infrastructure — not full marketing automation.

10. Decision Guidance For What Comes After

After terminal design and communications foundation, the next step should not be chosen randomly. Ask:

  1. What strengthens the payment-to-customer loop most?
  2. What makes Patronr more actionable, not just more descriptive?
  3. What builds on the new communications foundation naturally?
  4. What benefits most from the groundwork already laid?

See the full ROS build roadmap in section 14 for the recommended sequence.

11. Current Product Maturity Read

Right now Patronr should be understood as:

  • Strong in foundational architecture relative to its stage
  • Still early in visible module buildout
  • Intentionally focused on the parts that prevent future breakage

This is not shallow work. It is platform hardening and direction-setting work.

12. Current Strategic Summary

Patronr is still in groundwork mode. The current mission is not to rush module count.

Make the terminal strong · make payment-linked capture strong · make authority and branch rules stable · make communications foundationally correct · then build outward on stable ground.

13. Immediate Next Steps

The current working sequence:

  1. Terminal design completion
  2. Revenue classification layer — catalog items, areas, events, covers (schema + checkout UI + admin pages)
  3. Promotions and discounts
  4. SMS and WhatsApp foundation (co-primary channels for Kenya)
  5. Email foundation alongside only if it does not dilute focus
  6. Patronr API and Angavu integration for messaging
  7. Re-evaluate next strategic module using the ROS pillar framework
Channel priority: SMS and WhatsApp together first. Email is useful for reports and formal communications but should not hold up the core channels.

15. Priority Sequence For What Comes After Terminal

When terminal design is complete, the recommended build order:

  1. 1
    Revenue classification layer Enriches every transaction going forward and the build cost is low. Schema changes are easier to make now than after thousands of transactions have accumulated without this context.
  2. 2
    Promotions and discounts Revenue data is unreliable without it. Discounting outside the system creates dirty numbers that corrupt every downstream report.
  3. 3
    SMS and WhatsApp foundation The first action channel. Everything in phases 6 and 7 depends on this being right. WhatsApp is co-primary with SMS for Kenya.
  4. 4
    Customer segmentation and churn signals Makes all the data already being captured actually useful. Feeds directly into messaging and the manager action surface.
  5. 5
    Scheduled reports Low build cost, high daily value. Owners get a morning briefing without logging in.
  6. 6
    Automated triggers Builds on the communications foundation. Works silently and compounds over time.
  7. 7
    Manager action surface The interface that ties intelligence to action. Requires segmentation and communications to be in place. This is where Patronr feels like a Revenue Operating System.
  8. 8
    Campaign builder and outcome tracking Closes the ROS loop. Turns the intelligence and messaging layers into a measurable retention and revenue action tool.
This is not a rigid delivery schedule. It is the correct conceptual order. If something in a later phase becomes urgently needed, it can be pulled forward — but the dependency chain should be understood before doing so.

16. The Four ROS Pillars as a Build Test

Every feature built from here should be tested against the four pillars:

  • Capture   Does this bring more revenue context or customer identity into the system?
  • Understand   Does this turn existing data into useful intelligence?
  • Act   Does this enable a direct, fast, measurable action?
  • Measure   Does this close a loop and show whether something worked?
A feature that serves none of these four pillars should be questioned before being built.

17. Progress Rule

The system should not be judged only by how many pages exist, how many modules are visible, or how much UI has been completed.

It should also be judged by:

  • How much future breakage has been prevented
  • How much architectural confusion has been removed
  • How much operational correctness has been locked in
  • How much of the four-pillar ROS loop is functioning end to end
That is the stage Patronr is in now.
